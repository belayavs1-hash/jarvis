# Скилл: /write-plan

## Назначение

Превратить готовую концепцию в пошаговый план микрозадач. Каждый шаг — 2–5 минут работы, конкретный файл, конкретная проверка.

## Когда использовать

Концепция понятна (своя или из `/brainstorm`), но непонятно с чего начинать делать. Нужна дорожная карта.

## Как работает

### Шаг 1 — Собери контекст

Прочитай:
- Концепцию (из `memory/concept-*.md` или из разговора)
- `MEMORY.md` — стек, инфраструктура, ограничения
- `GOALS.md` — приоритеты и дедлайны

Если концепции нет — предложи сначала запустить `/brainstorm`.

### Шаг 2 — Составь план

Разбей проект на микрозадачи по принципу:
- Каждая задача = одно конкретное действие (не «настроить фронтенд», а «создать файл index.html с базовой структурой»)
- Время выполнения: 2–5 минут
- После каждой задачи — понятно как проверить что она сделана

Формат каждой задачи:
```
[ ] Номер. Название задачи
    Файл: какой файл создать/изменить
    Действие: что конкретно сделать
    Проверка: как убедиться что готово
```

Группируй задачи по этапам (не более 5 этапов).

### Шаг 3 — Сохрани план

Сохрани в `_progress.md` в корне проекта (или в папке проекта если она уже существует):

```markdown
# План: [Название проекта]

Дата: YYYY-MM-DD
Концепция: [ссылка или краткое описание]
Статус: в работе

## Этап 1: [Название]
[ ] 1.1 ...
[ ] 1.2 ...

## Этап 2: [Название]
[ ] 2.1 ...
...

## Завершено
> Сюда переносим выполненные задачи с датой
```

### Шаг 4 — Покажи план пользователю

Покажи план и спроси подтверждение. Если есть правки — внеси и покажи обновлённый вариант.

Когда план одобрен — предложи `/execute-plan`.

## Важные правила

- **Микрозадачи, не мегаблоки** — «настроить всё» не задача
- **Конкретный файл** в каждой задаче — не «что-то создать», а «создать src/index.js»
- **Проверка обязательна** — без неё непонятно когда задача закрыта
- **Не начинай делать** — только планировать. Выполнение — через `/execute-plan`
- Если проект уже существует — прочитай `git log` и `git status` перед составлением плана
